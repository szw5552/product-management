# PRD 評審細則 (Rubric)

在評審產品需求文件 (PRD) 時使用此細則。請內化這些標準，但以敘述性的方式提供反饋 — 而非僅僅是一個記分卡。產品經理 (PM) 在聽取反饋後應明確知道需要修復什麼以及為什麼，而不是盯着一張數字表格看。

---

## 評分標準

| 分數 | 意義 |
|-------|---------|
| **0 — 缺失** | 章節缺失或僅包含佔位符文本 |
| **1 — 薄弱** | 章節存在但缺乏實質內容、特異性或證據 |
| **2 — 合格** | 章節涵蓋了基本要點，但存在差距或可以更精煉 |
| **3 — 優秀** | 章節清晰、具體、有證據支持且具備可操作性 |

---

## 各章節細則

### 1. 問題陳述

| 分數 | 標準 |
|-------|----------|
| 0 | 無問題陳述，或僅僅是偽裝成問題的功能描述 |
| 1 | 描述了問題但很模糊，缺乏關於受影響者或嚴重程度的具體信息 |
| 2 | 問題清晰並識別了人物誌，但缺乏量化的影響或證據 |
| 3 | 問題具體，人物誌清晰且有嚴重程度評級，由定量及定性證據支持，並闡明了不採取行動的後果 |

**關鍵問題：** 你能否用一句客戶會認同的話來陳述問題？是否有證據表明這是真實的而非假設的？我們是否知道受影響的人數及其嚴重程度？

---

### 2. 目標與成功指標

| 分數 | 標準 |
|-------|----------|
| 0 | 未定義目標或指標 |
| 1 | 存在目標但很模糊（如「改善體驗」），且無可衡量的目標值 |
| 2 | 有帶目標值的可衡量目標，但缺失基準值、時間表或護欄指標 |
| 3 | 具備首要和次要目標，包含具體指標、當前基準值、目標值、衡量方法、評估時間表，以及不得退化的護欄指標 |

**關鍵問題：** 工程師能否根據這些指標明確知道需要埋點什麼？我們是否真的能在發布 4 週後判斷這項工作是否奏效？我們是否定義了「完成」的標準？

---

### 3. 解決方案概覽

| 分數 | 標準 |
|-------|----------|
| 0 | 未描述解決方案，或僅僅是沒有背景的功能列表 |
| 1 | 僅有高層次描述，無用戶流程、無原型圖、未考慮替代方案 |
| 2 | 方法清晰並包含用戶流程，但缺失原型圖/設計稿或未考慮替代方案 |
| 3 | 方法表述清晰且專注於用戶體驗，記錄了關鍵用戶流程，連結了原型圖或設計稿，考慮了替代方案並為所選方法提供了清晰理由，評估了「不採取行動」的選項 |

**關鍵問題：** 設計師能否根據此內容開始工作？這是否解釋了*為什麼*選擇此方法而非其他方法？解決方案是否圍繞用戶成果而非僅僅是系統行為構建？

---

### 4. 需求 (P0/P1/P2)

| 分數 | 標準 |
|-------|----------|
| 0 | 未列出需求，或僅是模糊的功能願望清單 |
| 1 | 存在需求但無優先級劃分，無驗收標準 |
| 2 | 劃分了需求的優先級 (P0/P1/P2)，但驗收標準模糊或不一致 |
| 3 | P0/P1/P2 層級清晰，每項需求都有具體且可測試的驗收標準，包含明確的「範圍外」章節並說明理由 |

**關鍵問題：** QA 能否僅根據驗收標準編寫測試用例？是否清楚哪些是發布阻礙因素，哪些是額外加分項？我們是否明確說明了我們*不*構建什麼？

---

### 5. 技術考量

| 分數 | 標準 |
|-------|----------|
| 0 | 未提供技術背景信息 |
| 1 | 技術備註極少，無風險評驗 |
| 2 | 有架構備註並識別了一些風險，但缺失遷移計劃或向後兼容性分析 |
| 3 | 包含數據模型變更、API 變更、第三方依賴、性能考量、帶緩解措施的技術風險、遷移及回滾計劃 |

**關鍵問題：** 工程團隊是否評審過此內容？這裡是否隱藏著任何「未知的未知數」？如果出現問題，回滾計劃是什麼？

---

### 6. 設計考量

| 分數 | 標準 |
|-------|----------|
| 0 | 無設計考量 |
| 1 | 簡要提到 UX，但無邊際情況或錯誤狀態 |
| 2 | 陳述了 UX 原則，識別了一些邊際情況，但錯誤狀態不完整 |
| 3 | 具備此功能的清晰 UX 原則，包含預期行為的全面邊際情況，以及帶有用戶提示訊息和恢復路徑的錯誤狀態 |

**關鍵問題：** 當出錯時會發生什麼？空白狀態是什麼樣的？對於擁有正常數據量 10 倍或 100 倍的用戶會如何？

---

### 7. 發布計劃

| 分數 | 標準 |
|-------|----------|
| 0 | 無發布計劃 |
| 1 | 計劃模糊（如「我們將逐步推廣」），無具體细节 |
| 2 | 定義了分階段推廣，但缺失發布清單或溝通計劃 |
| 3 | 推廣階段清晰，包含各階段的受眾、持續時間和成功標準；包含全面的發布清單（埋點、功能旗標、文檔、支持團隊簡報）；按受眾劃分的溝通計劃 |

**關鍵問題：** 埋點驗證是否在清單中？支持團隊是否知道即將發布？從 Beta 到 GA (正式發布) 之前必須滿足什麼條件？

---

### 8. 時間表

| 分數 | 標準 |
|-------|----------|
| 0 | 無時間表 |
| 1 | 僅有單一目標日期，無里程碑 |
| 2 | 有日期和負責人的里程碑，但未標註依賴關係 |
| 3 | 包含日期、負責人的里程碑，並識別了依賴關係及其延遲影響分析；包含發布後的成功評審日期 |

**關鍵問題：** 是否有任何可能阻礙我們的依賴項？誰負責每個里程碑？我們何時評估這項工作是否真的奏效？

---

### 9. 風險與緩解措施

| 分數 | 標準 |
|-------|----------|
| 0 | 未識別風險 |
| 1 | 列出了風險但無類別劃分、可能性評估或緩解措施 |
| 2 | 對風險進行了類別、可能性和影響的劃分，但緩解措施模糊 |
| 3 | 對風險進行了分類（產品/技術/市場/運營）並評估可能性和影響，具備具體的緩解計劃，並指派了負責人 |

**關鍵問題：** 我們是否在誠實面對可能出錯的地方？緩解措施是否真的降低了風險，還是僅僅將風險重複陳述為一個計劃？是否有人對每個風險負責？

---

### 10. 待解決問題

| 分數 | 標準 |
|-------|----------|
| 0 | 無待解決問題章節（這很可疑 — 永遠都有待解決的問題） |
| 1 | 列出了問題但無負責人或截止日期 |
| 2 | 有負責人和截止日期的問題，但有些內容陳舊或已經有答案 |
| 3 | 問題活躍且相關，具備負責人、截止日期，並跟蹤回答後的解決方案 |

**關鍵問題：** 這些是真正的未知數，還是產品經理正在迴避的決策？這些問題是否會在開發開始前解決？

---

## 准入基準 (Pass/Fail Criteria)

**PRD 在滿足以下所有條件前不得獲得批准：**

1. **清晰的問題陳述** — 具體、由證據支持、並識別了人物誌（第一章評分 ≥ 2）
2. **可衡量的目標** — 至少有一個帶有基準值和目標值的核心指標（第二章評分 ≥ 2）
3. **帶驗收標準的 P0 需求** — 每個阻礙發布的需求都有可測試的標準（第四章評分 ≥ 2）
4. **帶目標值的成功指標** — 已定義指標、衡量方法和評估時間表（第二章評分 ≥ 2）
5. **帶里程碑的時間表** — 至少有關鍵里程碑、日期和負責人（第八章評分 ≥ 2）
6. **風險評估** — 識別了主要風險並包含緩解措施（第九章評分 ≥ 1）

如果其中任何一項不符合要求，PRD 在進入開發階段前需要進行修訂。

---

## 如何使用此細則

**不要**直接給 PM 一張填好的計分卡。而是要：

1. 帶著這些細則閱讀 PRD
2. 找出最大的差距 — 到底缺失了什麼或哪裡薄弱會真正導致問題？
3. 以段落形式編寫反饋：先列出最關鍵的問題，解釋其重要性，並建議「好」的做法是什麼樣的
4. 表揚做得好的地方 — PM 需要知道什麼是奏效的，以便保持下去
5. 給出明確的結論：批准、需要小幅修訂（列出清單）或需要重大重做（說明原因）

**反饋範例：**

> 這份 PRD 有紮實的問題陳述，需求結構也很好 — 你顯然做過了客戶研究。主要的差距在於衡量方式：目前沒有定義成功指標，這意味着我們在發布後無法判斷它是否奏效。在交給工程團隊之前，你需要定義至少一個核心指標，包含當前基準值和一個目標值。我還建議增加頁面加載時間的護欄指標，因為這涉及核心工作流。時間表章節需要里程碑 — 目前只有一個目標發布日期，中間沒有任何檢查點。
>
> **結論：** 需要小幅修訂。請添加成功指標和時間表里程碑，完成後即可準備就緒。

*此細則是一種思考工具。其價值在於它所引發的批判性思考，而非分數本身。*
