# 優先級排序框架

我們如何決定要構建什麼。此框架確保我們將有限的資源投入到影響力最高的工作中。

---

## 哲學

優先級排序是產品團隊所做的最重要的事情。說「不」比說「是」更有價值。一個清晰的框架不會替我們做決定 — 它使我們的推理變得明確、可討論且可改進。

我們在三個層面進行優先級排序：
1. **戰略性：** 我們投資於哪些問題領域？（每季度）
2. **戰術性：** 我們承諾哪些具體倡議？（每週期）
3. **進行中：** 當週期中需要更改範圍時，該如何取捨？（持續）

---

## 第 1 層：戰略優先級排序（每季度）

### 投資組合

每個季度，我們將總產能分配到四個類別中：

| 類別 | 目標分配 | 描述 |
|--------|------------------|-------------|
| **新價值** | ~40% | 目前不存在的新功能和能力 |
| **改進現有** | ~30% | 根據客戶反饋和數據對現有功能進行增強 |
| **維持運作** | ~15% | 錯誤修復、維護、基礎設施、安全性 |
| **探索** | ~15% | 研究尖峰（Research Spikes）、原型、本季度可能不會有回報的戰略賭注 |

這些是指南，而非規則。確切的比例會根據產品成熟度、市場動態和戰略優先級而調整。但如果任何類別降至 0%，則說明出了問題。

### 戰略評分

為了決定投資哪些*問題領域*，我們評估：

| 標準 | 權重 | 分數 (1-5) | 描述 |
|----------|--------|-------------|-------------|
| **客戶影響** | 30% | | 有多少用戶受到影響？痛苦程度有多深？ |
| **戰略一致性** | 25% | | 這是否推動了我們的核心價值主張和長期願景？ |
| **收入影響** | 20% | | 這是否推動了獲客、留存、擴展或變現？ |
| **可行性** | 15% | | 我們能在一個週期內交付一個有意義的版本嗎？ |
| **學習價值** | 10% | | 即使它沒有「奏效」，我們是否會學到一些關鍵的東西？ |

**總分 = Σ (權重 × 分數)**

## 第 2 層：戰術優先級排序（每週期）

### 受 RICE 啟發的框架

為了決定哪些*具體倡議*能進入一個週期，我們使用修改後的 RICE 評分：

**影響分數 = (觸達範圍 × 影響力 × 信心) / 投入精力**

| 因素 | 規模 | 如何估算 |
|--------|-------|----------------|
| **觸達範圍 (Reach)** | 每季度受影響的用戶/帳戶數量 | 使用產品分析、細分市場規模數據 |
| **影響力 (Impact)** | 0.25 (極小), 0.5 (低), 1 (中), 2 (高), 3 (巨大) | 基於對目標指標的預期效果 |
| **信心 (Confidence)** | 100% (高), 80% (中), 50% (低) | 基於證據的質量 — 用戶研究、數據、過去的實驗 |
| **投入精力 (Effort)** | 人週工作量 | 工程估算（包括設計、QA、文檔編寫） |

### 優先級層次

| 等級 | 意義 | 行動 |
|------|---------|--------|
| **P0 — 必須做** | 阻礙收入、破壞核心 UX 或對戰略承諾至關重要 | 本週期必須交付，無例外 |
| **P1 — 應該做** | 對核心指標有很高影響力，有強大的客戶需求證據 | 如果產能允許，在本週期交付 |
| **P2 — 可以做** | 有意義的改進，證據程度中等 | 下一週期候選項目 |
| **P3 — 現在不做** | 有趣但影響力較低或證據不足 | 待辦清單，每季度重新評估 |

### 堆疊排序測試

評分後，對所有候選項目進行堆疊排序，並在您的產能處劃線。線上的一切都是承諾的。線下的一切明確為不承諾。

**規則：**
- 這條線是神聖的。不要試圖再「塞進一件小事」。
- 如果週期中出現了在線上的新事情，則當前在線上的一些事情必須移到線下。
- 向利益相關者展示完整的排名列表（包括線下的內容）。對於我們*不*做什麼的透明度與我們要開發什麼同樣重要。

## 第 3 層：進行中優先級排序（持續）

### 當週期中範圍發生變化時

計畫趕不上變化。出現了關鍵錯誤、重要客戶有緊急需求，或者某個假設被證明是錯誤的。當這種情況發生時：

**權衡對話：**
1. **命名新事物**並估算其投入精力。
2. **命名它取代的事物。** 必須有一些東西從清單中移除。「我們只要更努力工作」是不可接受的。
3. **取得一致意見**，來自 Pod 負責人和 PM。如果權衡意義重大，請讓領導層參與。
4. **溝通變更**給那些期待被取代工作的利益相關者。

### 升級階梯

| 情況 | 誰決定 |
|-----------|-------------|
| 功能內的範圍權衡（例如：削減邊緣案例） | Pod (PM + 工程主管) |
| 在同一個 Pod 內用一個倡議換另一個 | PM + 工程主管 |
| 將資源從一個 Pod 調配到另一個 Pod | PM + 領導層 |
| 更改季度戰略優先級 | 領導團隊 |

## 常見的優先級排序陷阱

### 要避免的陷阱

1. **聲音最大的客戶獲勝。** 一個客戶的尖叫 ≠ 一種模式。尋找來自多個來源的信號。
2. **HiPPO（高薪人士的意見）。** 高階主管的想法與其他任何事情一樣，都要經過相同的框架。
3. **沉沒成本謬論。** 「我們已經開發了一半」不是完成一件沒有價值的事情的理由。
4. **緊急 vs. 重要。** 救火感覺很緊急，但戰略性工作會創造複利價值。為重要但非緊急的工作保留時間。
5. **追逐功能對等（Feature Parity）。** 僅僅因為競爭對手有某個功能就去開發它，而沒有證據表明我們的客戶需要它。
6. **隱藏複雜性。** 「這只是一個小功能」通常會隱藏集成工作、邊緣案例和持續維護。

### 健康優先級排序的徵兆

- 您的待辦清單中標記為「現在不做」的項目多於「進行中」的項目
- 您可以向任何利益相關者解釋為什麼他們的要求不在當前週期中 — 並帶有細節
- 工程師對他們接下來要開發的東西並不感到驚訝
- 您說「不」的次數至少比說「是」多 3 倍
